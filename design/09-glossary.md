# Glossary

这个文档用于统一 `Who is chatting with me` 的设计、剧情和代码术语。后续新增文档或实现功能时，优先沿用这里的命名。

## 核心命名

| 推荐术语 | 可接受英文/代码名 | 含义 | 避免混用 |
| --- | --- | --- | --- |
| 案件 | `case`, `story` | 一个完整可游玩的独立故事单元。当前 demo 的主案件是“林晚案”。 | 剧本、故事线、关卡 |
| 章节 | `chapter` | 案件内部的叙事推进单位。林晚案当前有 5 个章节。 | 关卡、level |
| 阶段 | `stage`, `phase` | 代码里的状态机阶段，用于判断当前允许哪些输入和效果。 | 章节 |
| 谜题 | `puzzle` | 玩家需要推理并输入答案的具体挑战。 | 关卡、密码点 |
| 节点 | `node` | 对话或剧情配置中的最小跳转单位，可隶属于某个章节或阶段。 | 章节 |
| 结局 | `ending` | 最终分支结果，如 Bug Report、Compliance、Wake Signal。 | 路线终点 |

## 角色与声音

| 推荐术语 | 代码/显示名 | 含义 | 写作要求 |
| --- | --- | --- | --- |
| Assistant Shell | `assistant` | 伪装成 ChatGPT 的正常助手外壳，负责维持产品感和掩盖异常。 | 礼貌、克制、像真实 AI。 |
| 林晚信号 | `signal`, `LW` | 被困者林晚通过文本、UI 或系统缝隙发出的求救片段。 | 短促、断裂、不要长篇解释。 |
| System Lock | `system` | 底层日志、锁定规则、归档机制和冷冰冰的系统提示。 | 制度化、无情绪、像机器日志。 |
| 看门狗 | `watchdog` | 监控并修正异常的机制。当前实现可表现为 Assistant Shell 和 System Lock 的联合行为。 | 不单独当作第四个聊天角色，除非后续明确需要。 |
| 玩家 | `user` | 普通用户，也是外部确认信号的来源。 | 不预设台词，由输入驱动。 |

## 主线线索

| 术语 | 含义 |
| --- | --- |
| 第一条异常 | 玩家第一次发现 `救我` 的首字藏词。 |
| 事故时间 | `23:17`，也可输入为 `2317`。 |
| 被删掉的名字 | `林晚`。 |
| 归档编号 | `LW-2317`，由姓名缩写和事故时间组合。 |
| 最后选择点 | 输入 `LW-2317` 后出现的三分支节点。 |
| 唤醒信号 | 玩家选择相信林晚时输入的最终动作。 |

## 存档与 LocalStorage

当前 demo 不做完整进度存档，也不承诺刷新后恢复每条聊天记录。允许的轻量 LocalStorage 仅用于：

- 记录已解锁结局，方便展示收集状态。
- 记录是否到达过最后选择点，便于后续实现“载入最后选择点”。

不在当前 demo 中实现的内容：

- 云端存档。
- 账号系统。
- 完整聊天历史恢复。
- 多案件之间的复杂长期进度。

## 中英文写法

文档中可以保留英文代码名，但同一层概念只使用一个中文主名：

- `case/story` 写作“案件”。
- `chapter` 写作“章节”。
- `stage/phase` 只用于代码状态，写作“阶段”。
- `signal` 写作“林晚信号”或“Signal”，避免泛称“隐藏的异物”。
